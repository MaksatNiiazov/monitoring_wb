# Уточнения к реализации

Документ фиксирует уточнения по задаче после просмотра ТЗ, образца и расшифровки видео.

## Главная цель

Нужно автоматизировать текущую ручную работу команды по заполнению мониторинга WB.

Веб-интерфейс Django нужен не как замена таблице, а как:

- панель управления процессом;
- хранилище истории и исходных данных;
- место для запуска обновлений за дату;
- место для настройки товаров, РК, складов и параметров;
- интерфейс для получения аналитики и контроля результатов.

## Что считается целевым результатом

Система должна:

- получать данные из WB API;
- сохранять их в БД;
- автоматически формировать и обновлять мониторинг в табличном формате;
- строить эту таблицу по шаблону, который лежит в корне проекта;
- давать возможность принудительно пересобрать данные за нужную дату.

## Важная трактовка задачи

Нам не нужно просто сделать "новую систему отчётов".

Нужно:

- автоматизировать существующий рабочий процесс;
- сохранить веб-интерфейс как центр управления;
- автоматически заполнять таблицу мониторинга в структуре, близкой к шаблону;
- использовать БД как внутренний источник истины;
- использовать табличный мониторинг как результирующий слой.

## Что взять из ТЗ как обязательное

1. Рекламная статистика и общая воронка заполняются за предыдущий календарный день.
2. Остатки и остатки по складам заполняются за текущий календарный день.
3. Должна быть автоматическая ежедневная синхронизация.
4. Должна быть ручная синхронизация с возможностью перезаписать данные за выбранную дату.
5. Нужно хранить историю данных.
6. Нужно добавлять товары по `nmID / артикулу WB`.
7. Нужно добавлять рекламные кампании по `ID РК`.
8. Нужно собирать данные по зонам:
   - поиск
   - рекомендательные полки
   - каталог
9. Нужно хранить остатки по всем складам.
10. Нужно уметь отображать только выбранные склады.

## Что важно из видео

1. Автоматизируем именно их текущий процесс заполнения мониторинга.
2. Для каждого товара логика заполняется отдельно.
3. Исторически таблица заполняется по датам.
4. Есть "белые" значения, которые нужно заносить данными.
5. Есть расчётные значения, которые не являются первичным источником данных.
6. Остатки по складам должны тянуться все, но показываться только выбранные.
7. Нужен запуск за конкретную дату.
8. Веб-интерфейс должен использоваться для управления процессом и просмотра аналитики.

## Текущее архитектурное решение, которое сохраняем

Сохраняем:

- Django-приложение;
- БД как основной контур хранения;
- сервисы получения данных из WB API;
- историю метрик;
- ручной и автоматический запуск синхронизации;
- аналитические страницы в веб-интерфейсе.

## Что нужно изменить в следующей итерации

1. Перестроить слой выгрузки/таблицы:
   - не как абстрактный отчёт;
   - а как автоматическое формирование мониторинга по шаблону в корне проекта.

2. Привязать табличный слой к шаблонной структуре:
   - по блокам;
   - по датам;
   - по товарам;
   - по зонам.

3. Сделать веб-интерфейс основным инструментом управления:
   - запуск обновления;
   - обновление за дату;
   - выбор складов;
   - настройка товаров и РК;
   - просмотр истории и аналитики.

4. Не принимать решение "заменяем ли таблицу UI":
   - UI для управления;
   - таблица для итогового мониторинга.

## Рабочее правило на следующую реализацию

Если возникает выбор между:

- "сделать красиво внутри веба"
- и "точно автоматизировать их реальный процесс"

приоритет всегда у второго варианта.
